## **0\. Назначение и границы**

**Назначение:** Обеспечить **устойчивую работу без ручного героизма**, с автоматическим торможением при рисках.

**Границы:**

* Playbook v7 действует при масштабе **x1 (без роста)**.  
* Коммерция/экспансия — **вне** этого документа.  
* Любые изменения — только через раздел 7\.

---

## **1\. Архитектура управления**

OWNER  
└─ CEO  
   └─ COO  
      ├─ Производство  
      ├─ Fulfillment  
      ├─ Логистика  
      ├─ Платформа (MVP-1)  
      └─ People Ops

**Жёсткие запреты:**

* CEO не управляет сменами.  
* COO не принимает финансовые риски.  
* MVP-1 не принимает бизнес-решений.  
* Роль ≠ человек (временные совмещения допустимы).

---

## **2\. KPI-иерархия (коротко)**

CEO  
├─ Cash Buffer (≥30 дней)  
├─ CF ≥ 0  
└─ Штрафы MP ≤ X/мес  
   ↓  
COO  
├─ План 95–105%  
├─ Брак ≤2%  
└─ Авралы = 0  
   ↓  
Роли  
├─ Производство (выпуск/брак)  
├─ Fulfillment (SLA/ошибки)  
├─ Логистика (OOS/штрафы)  
└─ Платформа (доступность/ошибки)

**Правило:** у каждого KPI есть **STOP**. Без STOP — KPI не существует.

---

## **3\. Перегрузы и STOP (коротко)**

Сигнал → Порог → Авто-действие → Эскалация

**Примеры авто-действий:**

* Freeze изменений  
* Упрощение плана  
* Консервация ассортимента  
* Read-only для MVP-1

**Цель:** поймать перегруз **на уровнях 2–4**, не доводя до CEO.

---

## **4\. Сценарии системы**

S0 — Норма  
S1 — Упрощение (убрать вторичное)  
S2 — Консервация (только ядро)  
S3 — Аварийный (STOP роста)  
S4 — Полный STOP (OWNER)

**Правило:** сценарий переключает **COO**, подтверждает **CEO**. Назад — только после стабилизации KPI и снятия перегрузов.

---

## **5\. Эскалации и D-точки**

Операция → Мастер → COO → CEO → OWNER

**Когда эскалировать обязательно:**

* Сработал STOP  
* Повторный инцидент  
* Конфликт контуров  
* Финансовый риск

---

## **6\. MVP-1: данные и действия**

Источники → Расчёты → Индикаторы → Сигналы → Действия

**MVP-1 делает:**

* Считает KPI  
* Детектит перегрузы  
* Применяет авто-действия  
* Логирует всё

**MVP-1 НЕ делает:**

* Не решает «что выгоднее»  
* Не меняет стратегию  
* Не обходит D-точки

---

## **7\. Регламент изменений (Change Control)**

Запрос → Анализ риска → Решение → Окно изменений → Наблюдение

**Правила:**

* Изменения запрещены при S2–S4.  
* Один change за раз.  
* Откат обязателен.

Контроль

├─ Аудит перегрузов (еженедельно)  
├─ Аудит ролей (ежемесячно)  
└─ Аудит MVP-1 (ежемесячно)
